========================
BACKEND: REQUIRED ENDPOINTS
========================

Implement or ensure these endpoints exist and are wired to the new flow. Keep them secure and simple:

1) POST /api/auth/signup/start
Input: { firstName, lastName, email }
Behavior:
- Create or upsert a "pending signup" record (do NOT create full user session yet)
- Generate a 6-digit code
- Store hashed code + expiry (10 min)
- Email the code to the user
Return: { ok: true }

2) POST /api/auth/signup/verify-email
Input: { email, code }
Behavior:
- Validate code + expiry
- Mark email as verified for that pending signup
Return: { ok: true }

3) POST /api/auth/signup/set-password
Input: { email, password }
Behavior:
- Only allow if email verified
- Create the real user account (or finalize existing user record)
- Hash password
- Create session / login user
Return: { ok: true, user }

4) POST /api/onboarding/owner/setup
Auth required
Input: { industry, companyName, employeeRange }
Behavior:
- Create company
- Assign current user as Owner
- Save industry + employeeRange on company profile
Return: { ok: true, companyId }

5) POST /api/onboarding/employee/join
Auth required
Input: { inviteCode }
Behavior:
- Validate invite code
- Add membership + role
Return: { ok: true, companyId, role }

6) GET /api/onboarding/pricing-preview?employeeRange=...
Optional helper:
- Return plan name, price, and included bullets for Step 7
(If pricing is static in frontend for now, ok, but prefer backend-driven later.)

SECURITY REQUIREMENTS:
- Rate limit verification code requests per email/IP (basic throttling).
- Verification codes must be hashed in DB (do not store plaintext).
- Codes expire after 10 minutes.
- Resend cooldown enforced.
