Connect Apple App Store and Google Play subscription data to the private dashboard in STAGING only.

Context:
The private dashboard is working at:

staging-dashboard.ecologicc.com

Currently working:
- Campaign tracking links
- Referral/source attribution
- Subscribers dashboard
- Sources dashboard
- Accounts dashboard
- Stripe trial/subscription sync
- Full account deletion from Accounts
- Dashboard access controlled by DASHBOARD_ADMIN_EMAILS

Now I want Apple and Android subscription data to feed into the same dashboard.

Goal:
When a user subscribes through iOS Apple IAP or Android Google Play billing, the dashboard should update the matching growth_subscribers row just like Stripe does.

The dashboard should show:
- Platform: Apple or Google Play
- Plan
- Status: Free Trial / Active / Canceled / Expired / Unknown
- MRR
- Onboarded: Yes
- Source/campaign/referral code if they came from a campaign link

Important:
Build in STAGING only.
Do not break Stripe.
Do not break Apple IAP validation.
Do not break Google Play validation.
Do not break login, onboarding, jobs, billing, account deletion, or campaign tracking.

PART 1 — Find existing mobile subscription validation

Find the existing endpoint:

POST /api/subscriptions/validate

It should already support:
- platform: apple
- platform: google_play

The app already validates:
- Apple JWS / transaction
- Google Play purchaseToken + productId

Do not rewrite the whole validation system.
Just hook the successful validation result into growth_subscribers.

PART 2 — Apple sync to growth_subscribers

When Apple subscription validation succeeds:

Find the company/user for the validated subscription.

Then find the matching growth_subscribers row by:
1. companyId
2. userId
3. appleOriginalTransactionId
4. appleTransactionId

Update growth_subscribers:

- platform = apple
- plan = friendly plan label
- subscriptionStatus = mapped Apple status
- monthlyRevenue = 0 if trialing
- monthlyRevenue = plan price if active/paid
- appleOriginalTransactionId
- appleTransactionId
- trialStartedAt if applicable
- becamePaidAt if paid/active and not already set
- canceledAt if canceled/expired/revoked
- onboardingCompletedAt if company onboarding is completed

Status mapping:
- trial / trialing / intro offer trial → trialing
- active/paid → active
- canceled/revoked/expired → canceled or expired
- unknown → unknown

Friendly display labels:
- platform apple → Apple
- trialing → Free Trial
- active → Active
- canceled → Canceled
- expired → Expired

PART 5 — MRR rules

During free trial:
- monthlyRevenue = 0
- status = trialing
- Overview Trialing count increases
- Current MRR does NOT include this user

When active/paid:
- monthlyRevenue = plan amount
- status = active
- Overview Paid Subscribers increases
- Current MRR includes this user

When canceled/expired:
- monthlyRevenue = 0
- status = canceled or expired
- Overview Canceled increases if canceled/expired is counted there

Do not create duplicate growth_subscribers rows if one already exists for the same company/user.

If a user has campaign attribution:
- preserve sourceType
- preserve sourceName
- preserve campaignId
- preserve referralCode

Do not overwrite attribution.

Only update subscription fields:
- platform
- plan
- subscriptionStatus
- monthlyRevenue
- Apple/Google identifiers
- trial/paid/canceled timestamps

If no growth_subscribers row exists:
- If company exists but no attribution row exists, create a growth_subscribers row with:
  - companyId
  - userId
  - ownerEmail
  - companyName
  - sourceType = unknown
  - platform = apple or google_play
  - subscription fields
