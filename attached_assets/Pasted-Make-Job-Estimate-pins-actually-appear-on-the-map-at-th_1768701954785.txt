Make Job/Estimate pins actually appear on the map at their real addresses (geocode + cache + render)

CURRENT

- Blue dot works in full browser tab (good)
- BUT scheduled Jobs/Estimates are NOT dropping pins based on their addresses

GOAL
On Schedule → Map (for the selected day):
- Every scheduled Job with an address shows a GREEN pin at the correct location
- Every scheduled Estimate with an address shows a PURPLE pin at the correct location
- If multiple items share the same location, they cluster
- Clicking a pin shows an info card and can navigate to the item

ROOT PROBLEM (LIKELY)
We do not have lat/lng stored for jobs/estimates/customers, and/or geocoding is failing due to incomplete address strings or the map view isn’t using the results.

For BOTH jobs and estimates, determine the address source in this priority order:

A) If item has explicit address fields (recommended):
- addressLine1
- city
- state
- zip

B) Else fallback to customer address fields

Construct fullAddress like:
"${addressLine1}, ${city}, ${state} ${zip}"
If city/state/zip missing, use safe fallback:
"${addressLine1}, Long Island, NY"
(temporary fallback so geocoder can resolve)

2) ADD COORDS STORAGE (CACHE) — PICK ONE PLACE
Preferred cache location: CUSTOMERS
Add fields:
- customers.latitude (float)
- customers.longitude (float)

(If customers already have lat/lng fields, use them.)

3) BACKEND: CREATE/VERIFY GEOCODING ENDPOINT WORKS
Endpoint:
POST /api/geocode
Body: { address: string }
Response: { lat, lng, formattedAddress }

Use server env:
GOOGLE_MAPS_API_KEY

Must:
- return lat/lng for valid addresses
- return 404 for invalid addresses
- NEVER expose the server key to frontend

4) BACKEND: SAVE COORDS TO CUSTOMER AFTER GEOCODE
After successful geocode:
- Update customer.latitude/customer.longitude
- Optionally update customer.city/state/zip if missing using formattedAddress parsing (optional)
