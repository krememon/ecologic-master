Goals

Deactivated users cannot sign in (credentials or OAuth) until reactivated.

Existing sessions are revoked immediately on deactivation.

After (re)activation, the status badge and menu update without refresh.

1) Auth Guards (block login when INACTIVE)

Backend

Ensure we check status during every sign-in flow (credentials + OAuth) before issuing a session/JWT.

NextAuth-style example (adapt to your auth lib):

// [...nextauth].ts or auth controller
callbacks: {
  async signIn({ user }) {
    const fresh = await db.user.findUnique({ where: { id: user.id }, select: { status: true } });
    if (!fresh || fresh.status !== 'ACTIVE') {
      throw new Error('ACCOUNT_INACTIVE');
    }
    return true;
  },
  async jwt({ token, user }) {
    if (user) {
      const { status, tokenVersion } = await db.user.findUnique({
        where: { id: user.id },
        select: { status: true, tokenVersion: true },
      });
      token.status = status;
      token.tokenVersion = tokenVersion;
    } else {
      // On subsequent requests, enforce tokenVersion match
      const u = await db.user.findUnique({
        where: { id: token.sub! },
        select: { tokenVersion: true, status: true },
      });
      if (!u || u.status !== 'ACTIVE' || u.tokenVersion !== token.tokenVersion) {
        throw new Error('SESSION_REVOKED');
      }
    }
    return token;
  },
}


Credentials flow (if custom)

After verifying password, run:

const user = await db.user.findUnique({ where: { email }, select: { id, status, tokenVersion } });
if (user.status !== 'ACTIVE') return res.status(401).json({ code: 'ACCOUNT_INACTIVE' });


Why users could still log in now

Common causes:

The check runs against a stale user object from the previous login, not a fresh DB read.

Guard only exists in middleware for API routes, not in sign-in callback.

OAuth callback path doesn’t run the status check.

Action

Ensure the sign-in path always re-queries the user and blocks if status !== 'ACTIVE'.

2) Immediate Session Revocation on Deactivate

Backend (deactivate endpoint)

Transaction:

await db.$transaction([
  db.user.update({ where: { id }, data: { status: 'INACTIVE', tokenVersion: { increment: 1 } } }),
  db.session?.deleteMany?.({ where: { userId: id } }), // if you store sessions
]);


Middleware for all authed API routes/pages

After extracting user from token/session, re-check:

if (user.status !== 'ACTIVE') return 401 ACCOUNT_INACTIVE;
if (jwt.tokenVersion !== dbUser.tokenVersion) return 401 SESSION_REVOKED;


Frontend

If API returns ACCOUNT_INACTIVE or SESSION_REVOKED, clear auth and redirect to /signin with a banner.

3) Reactivation Path

Backend (activate endpoint)

await db.user.update({ where: { id }, data: { status: 'ACTIVE' } });


Do not change tokenVersion on activate. User must log in again.

Sign-in then succeeds because status is ACTIVE.

4) Badge Not Updating After “Activate” (UI cache fix)

Root causes to check:

The badge uses employee.isActive (boolean) but API returns status: 'ACTIVE'.

React Query/SWR cache isn’t invalidated after mutation.

Optimistic update writes the wrong shape / wrong key (e.g., Active vs ACTIVE casing).

Fix

Normalize status in one place:

const mapStatus = (s: 'ACTIVE'|'INACTIVE') => ({ isActive: s === 'ACTIVE', label: s === 'ACTIVE' ? 'Active' : 'Inactive' });


After successful Activate/Deactivate mutation:

await queryClient.invalidateQueries({ queryKey: ['employees'] });
await queryClient.invalidateQueries({ queryKey: ['employee', targetId] }); // if detail panel exists


If you keep optimistic updates, also patch the local cache:

queryClient.setQueryData(['employees'], (old:any) =>
  old?.map((e:any) => e.id === targetId ? { ...e, status: nextStatus } : e)
);


Ensure the badge component reads from employee.status (string) or employee.isActive (boolean) consistently across list + card + detail.

Also update the action menu

If status becomes ACTIVE, the menu item should switch to Deactivate, not Activate. Make it derive from employee.status rather than a local stale state.

5) Acceptance Tests

Auth

Deactivate user → existing session is 401’d on next call (or WS kick) and user is redirected to /signin.

Attempt login while INACTIVE → 401 with ACCOUNT_INACTIVE.

Activate → login succeeds.

UI

Activate/Deactivate shows toast and the chip flips immediately; refresh not required.

Employees list and detail view stay in sync.
