PART A — Auto-return to native app (NO “tap here”)
Root cause

We are still using a web return page (/stripe/return) that tries to deep link via a clickable link, but iOS in-app browser often blocks auto-redirects unless done correctly (and we’re not closing the browser from the native side).

Correct behavior required

When Stripe redirects after success/cancel, the native app should intercept it, close the Stripe browser, and navigate inside app.

Implementation Plan (do this exactly)
1) Use deep links in Stripe success_url/cancel_url

In backend Checkout Session creation:

success_url = ecologic://stripe-return?result=success&session_id={CHECKOUT_SESSION_ID}

cancel_url = ecologic://stripe-return?result=cancel

Do NOT use any http(s) return page for native.

2) Ensure iOS is configured to open ecologic://

In Xcode:

Target → Info → URL Types

Add scheme: ecologic

In Capacitor config, ensure nothing blocks it.

3) Handle appUrlOpen and close Browser immediately

In app bootstrap (main.tsx/app init):

Add App.addListener('appUrlOpen', ...)

If url includes stripe-return:

await Browser.close() (close the in-app Stripe browser)

call /api/subscriptions/status and refresh client subscription state

route user:

success → /jobs (or wherever)

cancel → /billing

4) Remove the “Returning to EcoLogic…” web page for native flow

We should NOT land on that page on native anymore.
If it still exists for web, that’s fine, but native must not use it.

5) Safety fallback

If appUrlOpen doesn’t fire (rare):

Keep a manual “tap here” link on the web return page, but native should never rely on it.
