T009 — Partial payments still show “$TOTAL unpaid/owed” on Payments page: fix by (1) proving webhook is firing + (2) making invoice owed/status computed from succeeded payments (server-side, source-of-truth), not a stale invoice field.

SYMPTOM
- Invoice total $350
- Paid $200 (Stripe actually charged $200)
- Payments page still says $350 unpaid/owed (and/or invoice pill “Unpaid”)
=> This means either:
A) Webhook is NOT updating the invoice/payment status in DB (most common), OR
B) Webhook updates, but the Payments/Invoices read endpoints are NOT using the paid amount to compute “owed/status” (they’re using invoice.total only), OR
C) Payment rows exist but are stuck as non-succeeded (e.g., “requires_payment_method/processing”) so your “sum succeeded” is 0.

WE WILL FIX THIS THE SAFE WAY:
1) Add instrumentation to prove webhook is hitting and payments are being marked succeeded.
2) Make owed/status computed server-side from SUM(payments where status='succeeded') so UI can never be wrong again—even if incremental updates fail once.
3) Update UI to display owed/status from those computed fields.

PART 2 — MAKE OWED/STATUS IMPOSSIBLE TO BE WRONG
Right now your UI is clearly not reflecting partials. We fix that by computing from payments.

D) Define canonical computed fields
For any invoice:
- total_cents
- paid_cents = SUM(payments.amount_cents WHERE invoice_id = ? AND status='succeeded')
- owed_cents = max(total_cents - paid_cents, 0)
- status:
    if owed_cents == 0 -> 'paid'
    else if paid_cents > 0 -> 'partial'
    else -> 'unpaid'

E) Implement server-side compute on every invoice fetch + list
Wherever you have:
- GET /api/invoices/:id
- GET /api/invoices (list)
- GET /api/payments page data endpoint (if it’s based on invoices)
Update queries to include paid_cents and owed_cents computed from payments.

Example approach (Postgres):
- LEFT JOIN payments p ON p.invoice_id=i.id AND p.status='succeeded'
- SELECT i.*, COALESCE(SUM(p.amount_cents),0) AS paid_cents,
         GREATEST(i.total_cents - COALESCE(SUM(p.amount_cents),0), 0) AS owed_cents
- GROUP BY i.id

If invoice list is used in multiple places, create a reusable DB helper:
- getInvoiceWithComputedTotals(invoiceId)
- listInvoicesWithComputedTotals(filters)

F) OPTIONAL: keep invoice.amount_paid_cents/status columns, but derive them from SUM on webhook
If you want to persist for speed:
On webhook success AND on manual payment:
- recompute paid_cents via SUM(succeeded) for that invoice (do NOT increment blindly)
- set invoice.amount_paid_cents = paid_cents
- set invoice.status based on owed
This avoids double-increment bugs and makes retries safe.
