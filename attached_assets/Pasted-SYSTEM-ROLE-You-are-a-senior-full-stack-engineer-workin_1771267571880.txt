SYSTEM / ROLE:
You are a senior full-stack engineer working inside the EcoLogic codebase.
Follow existing payment success flow patterns.
Do not use email links or public signing routes.
Implement an in-app “signature capture” modal that appears immediately after successful payment.
Preserve RBAC (“hidden not disabled”) and do not break existing payment logic.

TASK:
After a payment is successful, automatically pop up an in-app signature capture modal (no request link, no email).

CONTEXT:
User does NOT want an e-signature request workflow. They want the customer’s signature captured immediately on the device right after payment succeeds (like a “sign here” prompt at checkout). The signature should be saved to the job/payment record and viewable later in the job’s E-signature/Approvals area (or a new “Signatures” section if that already exists).

REQUIREMENTS:
- When a payment becomes successful (Stripe or “Mark Paid” if applicable), show a modal:
  - Title: “Signature Required” (or “Confirm & Sign”)
  - Body text: “Please sign to confirm payment and completion.”
  - Signature pad (draw)
  - Printed name field (required)
  - Buttons:
    - “Clear” (secondary)
    - “Save Signature” (primary)
    - Optional: “Skip” ONLY if a company setting allows skipping; default NO skip
- No email links. No public signing route. No tokens.
- Save the captured signature to the database linked to:
  - companyId
  - jobId and/or invoiceId
  - paymentId (preferred)
  - signedByName
  - signedAt
  - signatureImage (base64/png) OR stored file URL if you already have uploads/storage
- After saving:
  - Close modal
  - No success toast/popups (only errors if failure)
- Prevent duplicates:
  - If a signature already exists for that payment, do not prompt again.
- Company setting:
  - “Require signature after payment” ON/OFF (default OFF)
  - If ON, enforce signature capture after payment success.

FRONTEND CHANGES:
1) Identify the “payment success” moment in the UI:
   - Stripe checkout success callback OR payment confirmation modal closing
   - Manual mark-paid success
2) Add logic:
   - After payment success state is confirmed and paymentId is known:
     - If company setting requireSignatureAfterPayment is true
     - AND payment has no signature yet
     - THEN open `SignatureCaptureModal`
3) Create `SignatureCaptureModal` matching EcoLogic modal style (reference Select Customer):
   - Header + close X (close disabled if signature required and no skip)
   - Signature pad component:
     - Use an existing lightweight library if already installed (preferred),
       otherwise add one: `react-signature-canvas` (common) or similar.
   - Printed name input (required)
   - Clear button clears canvas
   - Save Signature button disabled until:
     - canvas has ink AND printed name not empty
4) On Save:
   - Convert signature to PNG:
     - `canvas.toDataURL("image/png")`
   - POST to backend endpoint with paymentId/jobId/invoiceId + printed name + image
   - On success:
     - close modal
     - refresh job/payment details UI so signature shows as “Captured”
   - On error:
     - show error toast/message (keep errors)

BACKEND CHANGES:
1) Add a table for stored signatures (if not already present):
   Table: `payment_signatures` (or `job_signatures`)
   - id
   - company_id
   - job_id (nullable)
   - invoice_id (nullable)
   - payment_id (required, unique)  // prevents duplicates
   - signed_by_name (text)
   - signature_png_base64 (text) OR signature_file_url (text)
   - signed_at (timestamp)
   - created_at
2) Add endpoint:
   - `POST /api/payments/:paymentId/signature`
     Body: { jobId?, invoiceId?, signedByName, signaturePngBase64 }
     Behavior:
     - Auth + company required
     - Verify payment belongs to company
     - If signature already exists for paymentId, return existing (do not create duplicate)
     - Save signature record
     - Return saved signature metadata (id, signedAt)
3) Add endpoint to check signature status (optional but useful):
   - `GET /api/payments/:paymentId/signature`
     - returns signature or null
   OR include signature status in existing payment fetch endpoint.

RBAC / PERMISSIONS:
- Capture modal should appear for whoever is completing the payment flow in-app (office user).
- Viewing signatures:
  - Owner/Admin: always
  - Other roles: follow your existing Documents/E-signature visibility rules
- Do not expose signature capture to roles that cannot take payments if your RBAC restricts payments.

EDGE CASES:
- Stripe success vs “settled”:
  - Trigger after your system marks it “Paid” (same place you update UI to Paid)
- Page refresh after payment:
  - If payment is already paid and setting is ON but signature missing,
    show a persistent banner “Signature required for this payment” with a button “Capture signature”
    (so it’s not lost if user navigates away).
- Mobile:
  - Ensure signature pad is large enough and scroll doesn’t interfere while signing.
  - Lock body scroll behind modal.

TESTING STEPS:
1) Turn ON “Require signature after payment”.
2) Complete a payment (Stripe test):
   - Confirm payment becomes Paid.
   - Signature modal opens immediately.
3) Draw signature + enter printed name → Save.
4) Confirm:
   - Modal closes
   - No success toast
   - Payment/job shows “Signature captured” (and can view the image)
5) Repeat:
   - Ensure no prompt if signature already exists for that paymentId.
6) Error test:
   - Attempt Save with empty name or blank canvas → button disabled.

DELIVERABLES:
- In-app SignatureCaptureModal triggered after successful payment (no email/link flow).
- Signature saved to DB linked to payment/job/invoice.
- Duplicate prevention (unique on paymentId).
- Clean UX: no success popups; errors only.
