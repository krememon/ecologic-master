PHASE 2 REQUIREMENTS
1) Real transfer execution path

Extend the existing Phase 1 preview pipeline so that when a payment succeeds for a subcontracted job, the system can create a real Stripe transfer only when all safety checks pass.

Safety checks required before transfer:

accepted subcontract agreement exists

accepted agreement snapshot contains valid locked financial values

subcontractor company exists

subcontractor company has stripe_connect_account_id

Stripe Connect status is payout-ready

no existing successful transfer/audit record already exists for this exact payment+agreement combination

payout amount is valid and greater than zero

payout amount does not exceed collected amount

no negative company share

platform payment actually succeeded

If any check fails:

do not transfer

write/update audit as failed/blocked with exact reason

log exact reason cleanly

2) Source of truth for amount calculation

Use the accepted subcontract agreement snapshot as the source of truth.

Do not recalculate from UI state.
Do not trust current editable job values.
Do not use random live fee fields if the referral was already accepted.

The transfer amount must come from the locked accepted agreement snapshot, with proper handling for:

Full payment

use full stored contractor payout amount

Partial payment

If invoice/payment collected is less than full job amount, support proportional payout from the stored snapshot.

Example:

locked job total at acceptance = $1,000

locked contractor payout = $200

collected payment = $500

contractor payout for this payment = $100

This must be deterministic and based on stored snapshot values.

Also protect against cumulative overpayment across multiple partial payments.

3) Transfer execution timing

Stripe webhook after payment success

I want one canonical payout execution path.
