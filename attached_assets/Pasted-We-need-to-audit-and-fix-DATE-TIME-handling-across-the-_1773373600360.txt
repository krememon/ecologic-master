We need to audit and fix DATE/TIME handling across the entire EcoLogic app so dates are shown correctly everywhere.

PROBLEM
Dates are incorrect in multiple places across the app. The screenshot is just one example, but this needs to be treated as a global date/time consistency issue, not a one-off UI bug.

GOAL
Ensure that throughout the app:
- dates are correct
- times are correct
- displayed dates match the real event timestamp
- no off-by-one-day issues
- no UTC/local timezone confusion
- no inconsistent formatting between screens

IMPORTANT
This is a GLOBAL audit/fix.
Do NOT just patch the one screen shown in the screenshot.

We need to inspect and standardize date handling across:
- web
- native/mobile
- list views
- detail views
- payment history
- invoices
- jobs
- schedule/events
- messages/notifications
- created/updated timestamps
- “today / yesterday” labels
- any relative time formatting
- any server-generated timestamps rendered on client

LIKELY ROOT CAUSES TO CHECK
Investigate all common date bugs:
- UTC stored in DB but rendered as local incorrectly
- local dates being converted twice
- ISO strings parsed with `new Date(...)` inconsistently
- date-only values being treated as midnight UTC and shifting day
- server sending one format, client assuming another
- native and web using different format helpers
- mixing `Date`, string, and timestamp number types
- formatting based on browser timezone vs app/user timezone
- “createdAt” being used where “paidAt/collectedAt/scheduledFor” should be used
- wrong source field shown in UI
- manual formatting code duplicated across screens

Check for:
- custom formatter helpers
- direct `new Date(...)` usage in components
- `toLocaleDateString`
- `toLocaleString`
- `Intl.DateTimeFormat`
- `date-fns`, `dayjs`, `luxon`, or any other library

3) STANDARDIZE DATE/TIME RULES
Create and apply clear rules:

- Timestamp fields:
  store as canonical timestamp and render in the user’s local timezone consistently

- Date-only fields:
  render as date-only without timezone shifting

- Relative labels:
  “Today”, “Yesterday”, etc. should be calculated consistently in local time

- Display format:
  use consistent formatting patterns across the app

Examples:
- date only: `Mar 13, 2026`
- date + time: `Mar 13, 2026 at 7:59 PM`
