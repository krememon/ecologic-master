WHAT THE LOGS SHOW

On web / preview origin, after Google popup starts, /api/auth/user keeps returning 401.

Important lines:

[auth/google][debug] host=...picard.replit.dev ... platform=popup
/api/auth/google is being hit correctly for popup flow
then repeated:
[MobileAuth] Bearer sessionId not found or expired in DB
[MobileAuth] Bearer resolved userId: email_...
User not found in database
GET /api/auth/user 401

This strongly suggests the WEB app is sending a stale/native Bearer auth token during normal web auth checks, causing /api/auth/user to go through the MobileAuth path instead of relying on the normal web session cookie.

====================
MOST LIKELY ROOT CAUSE

A stale nativeSessionId or mobile auth header is leaking into the WEB app / preview app.

So after popup completes, the web app may actually be trying to fetch /api/auth/user with:

Authorization: Bearer <nativeSessionId>

instead of:

normal browser session cookie auth

That would explain:

why MobileAuth logs appear during web sign-in
why user lookup is failing with email_...
why web stays unauthorized even if popup flow is otherwise okay
