# EcoLogic Construction Management Platform

## Overview
EcoLogic is a professional construction management platform designed for trade contractors to centralize job management, streamline subcontractor coordination, enhance client communication, and manage invoicing. The platform aims to modernize construction workflows, improve project oversight, and foster better stakeholder communication through a real-time, PWA-enabled web application. The business vision is to innovate construction project management by leveraging advanced technology and user-centric design to maximize efficiency and profitability for contractors.

## User Preferences
Preferred communication style: Simple, everyday language.
Design preferences: Bold, uppercase "EcoLogic" branding with wide letter spacing
Black Screen Fix: Global fetch interceptor in main.tsx auto-attaches Bearer token to all same-origin fetch calls. HTML/CSS base layer ensures background color is always visible before React mounts.
Authentication: Quad authentication options (Email/Password + Replit + Google OAuth + Apple Sign-In)
Registration: Enhanced form with password strength meter and show/hide toggles
User Flow: Automatic login and dashboard redirect after account creation
Logo: Custom water drop with leaf logo positioned above EcoLogic title

## System Architecture
EcoLogic is a multi-tenant web application. The frontend is built with React 18 (TypeScript, Vite, Tailwind CSS with shadcn/ui, TanStack Query, Wouter, React Hook Form). The backend uses Node.js with Express.js (TypeScript) and PostgreSQL with Drizzle ORM.

**UI/UX**:
- Responsive design with PWA support and dark mode.
- Accessible components primarily from Radix UI, incorporating custom "EcoLogic" branding and logo.

- **Growth-Subscribers Multi-Platform Sync**: `growth_subscribers` is the all-in-one subscriber table behind every dashboard page (Subscribers / Sources / Platforms / Overview). Three sync entrypoints all live in `server/dashboard/storage.ts` and follow the same upsert-or-create pattern (match by companyId â†’ userId â†’ platform-specific transaction id; preserve any existing first-touch attribution; backfill empty owner/company/signup fields on hit; on miss CREATE a new row with `sourceType = "unknown"` and `platform` set to the originating platform). MRR rules are identical across platforms: trial = `$0`, active/past_due = plan price from `shared/subscriptionPlans.ts`, canceled/expired = `$0`. Lifecycle stamps (`trialStartedAt`, `becamePaidAt`, `canceledAt`) are written once and never overwritten. (1) **Stripe** â€” `syncStripeSubscriptionToGrowthSubscriber` is invoked from the existing webhook hooks in `server/index.ts` (`customer.subscription.{created,updated,deleted}`); unattributed signups now produce a `sourceType=unknown / platform=stripe` row instead of being silently dropped. (2) **Apple** â€” `syncAppleSubscriptionToGrowthSubscriber` is invoked from the Apple branch of `POST /api/subscriptions/validate` after `storage.updateCompany` succeeds; matches on `appleOriginalTransactionId` / `appleTransactionId` and writes `platform=apple`. Status mapping via `mapAppleStatusToGrowth` (Apple `subscriptionStatus` 1=active, 2=expired, 3=past_due, 4=expired, 5=canceled). (3) **Google Play** â€” `syncGoogleSubscriptionToGrowthSubscriber` is invoked from the Google branch of the same validate endpoint; matches on `googlePurchaseToken` / `googleOrderId` and writes `platform=google_play`. Status mapping via `mapGoogleStatusToGrowth` using Google `paymentState` + `autoRenewing` (paymentState=2 â†’ trialing, paymentState=1 + autoRenewing â†’ active, paymentState=1 + !autoRenewing â†’ canceled, etc). All three sync paths run inside best-effort try/catch so a sync failure never breaks the underlying validate / webhook response. Logging prefix `[growth-dashboard]` for every create/update.

- **Account Deletion (Dashboard)**: Hard-delete of a customer company is exposed through the Accounts drawer at `staging-dashboard.ecologicc.com` and gated by **four** layers: (1) `requireAuth + requireDashboardAdmin` route gate, (2) env-var `ALLOW_DASHBOARD_ACCOUNT_DELETION=true` (default OFF â€” staging-only), (3) confirmation payload `{ confirmText: "DELETE", understood: true }` + matching modal that disables the submit button until the admin types `DELETE` AND ticks the "I understand" checkbox, and (4) hardcoded protected list inside `server/adminDeleteService.ts` (currently `pjpell077@gmail.com` / company 415). Self-delete (admin is a member of the target company) is also blocked unless `ALLOW_SELF_ACCOUNT_DELETE=true`. The deletion runs `deleteCompanyDeep()` inside a single `db.transaction`, wiping ~33 tables â€” including `growth_subscribers` and `growth_account_admin` â€” and then performing **full auth-record cleanup** so the deleted account's email becomes immediately reusable for re-signup. Auth cleanup classifies every candidate user (company owner + every `company_members.user_id`) into one of three buckets â€” *willDelete* (orphan, not protected), *willKeepBecauseOtherCompany* (still belongs to another company), *willKeepBecauseProtected* (email is in `PROTECTED_OWNER_EMAILS` or the runtime `DASHBOARD_ADMIN_EMAILS` allow-list, or the user is the actor running the delete) â€” then for the *willDelete* bucket removes rows from `push_tokens`, `login_challenges`, `pending_signups` (matched by `LOWER(email)` since this table has no `user_id`), best-effort `sessions` (substring match on the JSONB `sess` column scoped to those user-ids), and finally `users`. The same classification runs read-only inside `previewAccountDeletion()` so the confirmation modal can show admins exactly which user accounts will disappear, which will be kept (and why), and how many `pending_signups` rows will be cleared, **before** they type DELETE. Stripe / Apple / Google subscriptions are **NOT** auto-canceled â€” local references are removed and the modal warns the admin to cancel external billing separately. Every delete writes an `admin_audit_logs` row (`actorEmail`, `targetType=company`, `targetId`, `targetName`, `beforeValue` JSON containing per-table counts + tablesAffected + `deletedUserEmails` + `keptUsers` + `authIdentitiesDeleted` + `sessionsDeleted` + warnings). Endpoints: `POST /api/admin/dashboard/accounts/:companyId/delete-preview` (read-only counts + per-user breakdown for the modal) and `DELETE /api/admin/dashboard/accounts/:companyId` (the destructive call). Logging prefix `[dashboard-accounts] â€¦`.
